# Phase 21 Requirements — Add `jentic-api-improve` skill and agent

## Scope

Port the `jentic-api-improve` agent skill and its companion subagent definition out of the private `jentic-skills-internal` repository and into this repository as public, Apache-2.0 OSS, distributed and documented exactly the way the existing `jentic-api-scorecard` skill already is.

The skill teaches a coding agent to raise an OpenAPI document's JAIRF score: it runs a baseline `score --with-llm --detail diagnostics` via `@jentic/api-scorecard-cli`, identifies weak dimensions and actionable `POOR_OPERATION_SEMANTICS` diagnostics, applies non-breaking improvements (adding `summary`/`description`/`example`/`tags`, never changing existing contracts), and emits three artifacts — an improved spec (always JSON), an OpenAPI Overlay 1.1.0 (the reusable, non-breaking delta), and a changelog with before/after scores. For multi-iteration loops the skill spawns the companion `jentic-api-improve` subagent; where that subagent is not installed it degrades to spawning a subagent from an inline brief.

Concretely this phase lands the skill at `skills/jentic-api-improve/` (SKILL.md + six `references/` files) and the subagent at a new repo-root `agents/jentic-api-improve.md`, then wires both into every distribution and documentation surface that the scorecard skill uses: the Claude Code plugin marketplace (`.claude-plugin/marketplace.json`), the npm tarball (`packages/cli/package.json` `files` + `prepack`/`postpack`), the Vercel `skills` CLI and TanStack Intent (which need no change beyond the files existing), the `README.md`, the docs.jentic.com publish config (`docs/publish-config.json`), the architecture doc (`docs/architecture.md`), and the project instructions (`.claude/CLAUDE.md`).

## Out of Scope

- **Rewriting or restructuring the skill/agent content.** The source files are already written against the public `@jentic/api-scorecard-cli` and reference the repo-root `agents/jentic-api-improve.md` location that this phase creates — they are copied verbatim. The only permitted edits are corrections to any path or cross-reference that is wrong in this repo's context (verify there are none before editing).
- **Deduplicating the skill's `references/` against the existing scorecard skill or the README.** `references/jentic-api-scorecard.md` and `references/jentic-apitools-cli.md` restate CLI surfaces that also live elsewhere; they ship verbatim as self-contained, lazily-loaded references. Keeping them in sync with the CLI surface over time is a separate maintenance concern (a possible future extension of `.claude/rules/cli-readme-sync.md`), not this phase.
- **Deleting the skill/agent from `jentic-skills-internal`.** This phase only copies; the source removal is a separate follow-up the user owns.
- **Any change to the CLI surface, the Docker image, the runner, or the scoring engine.** The skill is markdown-only; it orchestrates the already-published CLI and external user-installed tools. No `packages/cli/src/`, `docker/`, `action/`, or formatter code changes, and therefore no `## CLI reference` / scorecard-`SKILL.md` flag-table sync (`.claude/rules/cli-readme-sync.md`).
- **Adding the external tools the skill orchestrates as repository dependencies.** `jentic-openapi-tools`, `jentic-apitools-cli`, and `check-jsonschema` are runtime prerequisites the end user installs (`pipx install …`) on their own machine to *run* the skill. They are not imported by any code in this repo, never appear in `package.json` / `docker/pyproject.toml`, and therefore do not trigger a `specs/tech-stack.md` update under `.claude/rules/update-tech-stack-on-deps.md`.
- **Weakening the skill to pass the SkillSpector security gate.** If `skill-security.yml` flags the new skill below a `SAFE` recommendation, the finding is surfaced to the user; the skill's documented safety design (its "Forbidden Shell Idioms" section) is not stripped to game the gate.

## Decisions

### Ship the agent via the Claude Code plugin *and* the npm tarball; degrade gracefully elsewhere
The companion subagent is inherently Claude-Code-specific — verified against the Claude Code docs, a `marketplace.json` plugin entry accepts an `agents: ["./agents/x.md"]` array of file paths, but the Vercel `skills` CLI and TanStack Intent are skill-only ecosystems with no concept of an agent. So the agent ships two ways: declared in the new plugin's `agents[]` (Claude Code installs it, registering it under `/agents`), and copied into the published npm tarball (so the SKILL.md's "copy `agents/jentic-api-improve.md` to your `.claude/agents/`" instruction resolves for npm/Intent consumers). Where the agent is absent, the skill's documented inline-brief fallback spawns an equivalent subagent — so no distribution path is broken, the plugin path is just the most convenient.

### Keep all source frontmatter verbatim
The source `SKILL.md` carries `license`, `metadata`, `compatibility`, `argument-hint`, and `allowed-tools`; the source `agents/jentic-api-improve.md` carries `model: inherit`, `tools`, and a (hyphenated) `allowed-tools`. Claude Code recognizes `argument-hint` and `allowed-tools` on a skill (the latter pre-approves the Bash patterns the skill relies on for prompt-free autonomous runs, which the skill is explicitly engineered around) and ignores the rest; on a subagent it recognizes `tools` and `model` and ignores `allowed-tools`. The unrecognized fields are harmless and document provenance and the skill's external-tool compatibility contract, so every field is preserved as-authored rather than trimmed to the repo's minimal `name`+`description` convention.

### A separate Claude Code plugin entry, not an addition to `api-scorecard`
`docs/architecture.md` §4 documents the growth path: a second *skill* on the same plugin is one more `skills[]` entry, but a distinct *plugin* gets its own `marketplace.json` entry. Improvement is a separable product from scoring (it carries extra runtime prerequisites — `jentic-openapi-tools`, `jentic-apitools-cli`, `check-jsonschema` — that a user who only wants a score should not be asked to install), and only the improve plugin should carry the agent. So `marketplace.json` gains a second plugin, `api-improve`, with its own `skills: ["./skills/jentic-api-improve"]` and `agents: ["./agents/jentic-api-improve.md"]`, installed independently via `/plugin install api-improve@jentic-api-scorecard`. The existing `api-scorecard` plugin is untouched.

### A new dedicated README section and a new docs.jentic.com page
Rather than fold the improve skill into the existing `## Agent Skills` section, it gets its own README section and its own `docs/publish-config.json` page — mirroring how the CLI, Action, and Enterprise topics each get a dedicated page. The new section documents the improve-specific prerequisites and the three install paths (Claude Code plugin `api-improve`, Vercel `--skill jentic-api-improve`, TanStack via the npm tarball) plus the agent note; it may cross-reference `## Agent Skills` for shared install mechanics rather than duplicating all of it. The section heading text must match the `sections[].heading` declared in `publish-config.json` exactly, because `scripts/extract-docs.js` slices the README by H2 heading.

### Mirror the existing `skills/` packaging mechanics for the new `agents/` tree
The npm tarball already carries `skills/` via the `files` array plus the `prepack` `copyfiles -u 2 "../../skills/**/*" .` / `postpack` `rimraf skills` pair (git keeps one canonical copy at the repo root; the tarball gets a real copy). The repo-root `agents/` tree is new, so it gets the symmetric treatment: a `"agents/"` entry in `files`, a `copyfiles -u 2 "../../agents/**/*" .` clause appended to `prepack`, `agents` added to the `postpack` `rimraf`, and `packages/cli/agents/` added to `.gitignore` (the staged copy, like the existing `packages/cli/skills/` ignore). The second skill needs no `files` change — `skills/` already globs it in.

## Constraints

Load-bearing invariants from `specs/mission.md`, `specs/tech-stack.md`, and `docs/architecture.md` that this phase must preserve.

- **`skills/` stays the single canonical source consumed by three distribution paths** (`docs/architecture.md` §4). The new skill must satisfy all three with no per-path duplication: the plugin reads it from the repo, Vercel reads it from the repo, and the npm tarball carries it via the existing `skills/` `files` glob + `prepack` copy. Do not introduce a second copy of the skill anywhere.
- **The Claude Code plugin marketplace reads the repo directly, not the tarball, and `.claude/` never leaks to installers** (`docs/architecture.md` §4). The new plugin entry uses `source: "./"`, `strict: false`, and explicit `skills[]`/`agents[]` arrays; it names only `./skills/jentic-api-improve` and `./agents/jentic-api-improve.md`, never anything under `.claude/`.
- **SkillSpector gates every `skills/*` directory on a `SAFE` recommendation** (`.github/workflows/skill-security.yml`). The workflow globs `skills/*` into its matrix, so the new skill is scanned automatically with no workflow edit — and the PR's CI will fail if the scan does not return `SAFE`. The repo-root `agents/` file is outside `skills/` and is not scanned.
- **`docs/architecture.md` is canonical, not aspirational** (`.claude/CLAUDE.md` "Open scope"). Adding a second skill, a new repo-root `agents/` directory, and a second plugin entry are architectural facts; the architecture doc's §4 layout tree and distribution notes are updated in the same PR.
- **`.claude/CLAUDE.md` must match what is on disk** (its own closing instruction in the `skills/` and repo-structure sections). The `skills/`, `.claude-plugin/`, and packaging descriptions are updated, and a new `agents/` entry added, in the same PR.
- **No mocking; tests hit real boundaries** (`.claude/rules/testing.md`). This phase ships no code, so no test *suite* runs — but the equivalent real-boundary checks are: the SkillSpector scan, an `npm pack --dry-run` proving the tarball actually carries the skill and agent, JSON validity of the edited config files, and a `docs:extract:dry-run` proving the new docs page generates.
- **Branch → atomic commits with DCO sign-off → PR → squash-merge; Conventional Commits** (`.claude/rules/git-workflow.md`, `.claude/rules/conventional-commits.md`). The work is grouped into atomic commits (files+packaging, marketplace, docs, constitution/lifecycle) each with a ≤69-char header.

## Context

This phase exists now because the improve skill has matured in `jentic-skills-internal` (its last source commits added flat output, end-of-run cleanup, and `verify-improvement` overlay verification) and the natural public home for an OpenAPI-improvement skill built entirely on top of `@jentic/api-scorecard-cli` is this repository, alongside the scoring skill it depends on. Scoring tells a user *what* is wrong with their API's AI-readiness; the improve skill closes the loop by *fixing* it and emitting a reusable, non-breaking Overlay — a direct extension of the product's mission.

The skill is the second consumer of the published scorecard CLI (the first being the CLI's own users): it calls `npx @jentic/api-scorecard-cli@latest score … --with-llm --detail diagnostics` for its baseline and every re-score. It is also the public counterpart to the `jentic-apitools verify-improvement` command and the OpenAPI Overlay format, both of which it documents in its `references/`. Because it is markdown-only and orchestrates already-shipped, user-installed tooling, the port is overwhelmingly a wiring-and-documentation exercise rather than a code change — which is why the validation below centers on distribution surfaces (tarball contents, marketplace entry, docs pages, the SkillSpector gate) rather than on a test suite.

`docs/architecture.md` (§4 layout + distribution notes) and `.claude/CLAUDE.md` (`skills/`, `.claude-plugin/`, packaging, and a new `agents/` description) are the two canonical documents updated in lockstep with the on-disk change.

## Stakeholder Notes

- **OpenAPI spec authors** — want to act on a low JAIRF score, not just read it. The improve skill turns a scorecard into concrete, non-breaking edits plus a reusable Overlay and a before/after changelog, satisfied by shipping the skill where their agent can discover it.
- **Claude Code users** — get the smoothest path: `/plugin install api-improve@jentic-api-scorecard` installs both the skill and the companion subagent, and the `allowed-tools` frontmatter lets the documented autonomous run proceed with minimal permission prompts.
- **npm / TanStack Intent / Vercel users** — get the skill through the channel they already use; the npm tarball additionally carries the agent file so the SKILL.md's copy-the-agent instruction resolves, and where the agent is not wired in the skill's inline-brief fallback keeps multi-iteration improvement working.
- **Security reviewers** — the new skill passes the automated SkillSpector `SAFE` gate before merge, and the skill's "Forbidden Shell Idioms" design (which keeps every shell invocation matched to an allowlisted pattern with no obfuscation surface) is the substance behind that verdict.
