# Phase 22 Retrospective Draft — Benchmark jentic-api-improve Token Usage and Cost

> **DRAFT** — written by `/sdd-implement-spec` from tracked implementation deviations. Promote to `retrospective.md` (edit + rename) or delete before merge if nothing here is worth capturing.

## Deviations from the spec

- **Core engine-measurement approach superseded (post-merge).** The whole spec (`requirements.md` Decisions, `plan.md` Group 1, `validation.md` check 5) was built around a net-new man-in-the-middle proxy (`scripts/token-proxy.mjs`) to capture the scoring engine's `--with-llm` tokens, precisely because "the scorecard JSON does not expose token usage" (`requirements.md:22-23`). After the spec merged (PR #285), `jentic-apitools` PR #253 added a native `tokenUsage` field to the scorecard for `--with-llm` score operations, resolving `specs/roadmap.md:339`'s open question ("confirm whether the scorecard output already surfaces usage") to **yes**. A follow-up change (PR #286) therefore bumped the engine to `1.0.0a25`, set `report_token_usage` on the runner's `OASProcessConfiguration`, had the improve skill write an aggregated `token-usage.json` output artifact, reworked the benchmark harness to read that field, and **deleted the proxy entirely** (`scripts/token-proxy.mjs` + its eslint scope). The merged `requirements.md`/`plan.md`/`validation.md` remain as the immutable record of what shipped in #285; this retrospective is the record of the reversal. Net effect: simpler design, no proxy, no `PROXY_UPSTREAM_*` setup — the engine tokens now come from the tool itself.

  A subsequent review-driven correction (same PR) made `tokenUsage` **opt-in** rather than default-on: the first cut hardcoded `report_token_usage=with_llm`, so every `--with-llm` score — including ordinary CLI and improve-skill runs for all users — emitted `tokenUsage`, polluting the public output. It is now gated behind a **hidden, benchmark-only `--report-token-usage`** CLI flag threaded CLI → container → runner (default off); the improve skill emits `token-usage.json` and passes the flag only when the invoker explicitly requests engine token usage (a "Report engine token usage" brief field the benchmark sets). Also removed the committed `scripts/bench-improve.data.json` (its engine fields were all-null — a confusing half-measured tracked artifact) and gitignored it; the doc now renders from an all-null dataset as "Not yet measured", and a real `npm run bench:improve` recreates the data locally. Lesson worth carrying: when adding an engine field that the public CLI streams verbatim, default it OFF and gate it behind an explicit opt-in — a field useful only to one internal caller must not change the default output for everyone.

  A further review-driven follow-up (same PR) added **run-outcome capture**: the harness measured token cost but never recorded the improvement outcome, so its Results table's baseline/score-after/iterations columns were always empty. The opt-in was generalized from "Report engine token usage" to "Report benchmark metrics", and the skill now also emits a structured `benchmark-summary.json` (`scoreBefore`/`scoreAfter`/`iterationsRun` + level/grade before+after) read via `jq` from the baseline and final scorecards — the same opt-in gate as `token-usage.json`, but not `--with-llm`-gated since the outcome exists on any run. The harness reads it (`readBenchmarkSummary`) and the Results table collapsed to `Score before` / `Score after` / `Iters`, dropping the always-null per-spec `Baseline score` column. Lesson: a benchmark should capture the *outcome* it's measuring the cost of, not just the cost — and structured JSON from the tool beats parsing a human-facing changelog.

  The **first real measured run** (petstore, `--with-llm`, opus/sonnet/haiku) then exposed a deeper class of problem the harness had been faithfully recording rather than catching. Benchmark end-scores came in lower (opus 66.38, sonnet 67.63) than an interactive run of the same skill on the same spec (68.8–70.73). Diagnosis against the actual shipped specs (not a measurement bug — the harness read the scores correctly): (1) opus added a `description` sibling to a bare `$ref` node, tripping `no-$ref-siblings` and cratering FC 99.47→74.10; it still *shipped* because the overall score rose from gains elsewhere, and its own local `jentic-openapi-tools validate` reported clean — only the engine re-score's FC dimension revealed it. (2) sonnet added operation-level `security` to 19 operations (a breaking change the skill forbids), because the SEC dimension strategy told it to ("sensitive operations must have security requirements") in direct contradiction of the non-breaking promise — the ARAX/DXJ strategies had the same latent trap for operationId/response-code additions. (3) a haiku cell ran without `--with-llm` on a non-comparable baseline, scored *lower*, and emitted a `benchmark-summary.json` with the wrong keys → read as null. The skill also defined the shipped/"after" spec as the highest-*numbered* iteration, not the best-*scoring* clean one, so a regressed iteration 2 was reported over a better iteration 1. Two fixes shipped (same PR): the skill/agent gained a **pre-ship no-dimension-regression guard** (revert to a best-clean-pass snapshot when any dimension drops below baseline — mirroring the corrective re-score the interactive run did to recover FC), an explicit `$ref`-sibling and operation-level-`security` prohibition in Constraints, resolution of the SEC/ARAX/DXJ strategy contradictions (name the ideal, apply only the additive subset, report the rest), and a redefinition of the shipped spec + `benchmark-summary.json` "after" as the best clean pass; the harness moved to **N samples per cell** (default 3, `--samples N`) reporting the median `scoreAfter` + min–max range and mean cost over *valid* samples only, with a `validateSample` predicate that flags non-comparable samples (no-`--with-llm`, regressed, malformed summary) as **errors** listed under "Excluded samples" rather than counted as measurements. Lessons: a benchmark must **validate comparability** before it records a number (a lower score can mean a broken sample, not a worse model); a strictly-additive tool must guard **per-dimension**, not just overall (an overall gain can hide a dimension regression); a skill's dimension-improvement guidance must not contradict its own non-breaking contract; local `validate` clean ≠ engine FC clean (`no-$ref-siblings`), so the tool cannot rely on validation to catch a self-inflicted regression.
- `plan.md` (footprint) / `validation.md` check 1 — the spec scoped the change to `scripts/` + `docs/` + `package.json` + `specs/roadmap.md`, but the lint gate required editing `eslint.config.js`: the repo globally ignores `**/*.js` / `**/*.mjs`, so `npx eslint scripts/…` reported "file ignored" (a no-op) rather than linting. A scoped config block (un-ignoring exactly `scripts/bench-improve.js` + `scripts/token-proxy.mjs` and giving them Node globals) was added so the gate is real. Scoped to those two files so the pre-existing, never-linted `extract-docs.js` was not retroactively pulled in.
- `plan.md` Group 5 — the real model×spec measurement was **not** performed (it needs Docker, a `JENTIC_API_KEY`, an authenticated `claude` CLI, and upstream LLM credentials, and spends real quota + budget). Per Group 5's explicit permission ("record a documented partial run and clearly mark unmeasured cells — never fabricate numbers"), the committed `docs/improve-cost-benchmark.md` was generated from an all-`null` data file and carries a "Not yet measured" banner with `—` placeholder cells. The renderer gained an unmeasured-aware banner path to support this.
- `plan.md` task 4 / `requirements.md` (input-spec set) — the spec targeted "2–3 more" OAK specs beyond the petstore anchor spanning low/mid/high baseline; the implementation pinned only the confirmed anchor petstore. Consistent with the unmeasured partial-run state (no baseline scores recorded yet), but the matrix is narrower than the target and should be widened when the real measurement is run.
- `validation.md` check 5 / `plan.md` tasks 3, 7 (found in pre-push review, fixed on-branch) — the first cut of `--dry-run` only printed static strings, so it did not exercise the proxy self-check or assemble the real env / `claude -p` plumbing the spec's Definition of Done requires. A fix-up commit rewired `--dry-run` to run the canned-upstream proxy self-check (asserting the tally) and to assemble+print the real per-cell plumbing. The Verify pass had initially been reported green on the strength of testing the proxy directly rather than through the dry-run path the spec mandates.
